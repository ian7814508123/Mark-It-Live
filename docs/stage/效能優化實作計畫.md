# 效能優化實作計畫：Markdown Live Previewer

根據 PageSpeed Insights 的反饋與資深工程師的建議，本計畫旨在透過程式碼分割、延遲載入及渲染優化，顯著提升應用程式的初始載入速度與交互流暢度。

## 🎯 優化目標
- **JS 載入量**：減少初始 Bundle 大小約 50-60%。
- **TBT (Total Blocking Time)**：降低主執行緒阻塞時間，提升響應速度。
- **效能分數**：從 40-50 分提升至 80 分以上。

---

## 🛠️ 第一階段：程式碼分割與延遲載入 (Lazy Loading)

目前許多大型套件（如 `mermaid`, `mathjax`, `vega`, `xlsx`）都在頂層直接匯入，導致初始加載過重。

### 1.1 圖表與數學公式延遲載入
- **Mermaid/Vega/Smiles/Abc**：
  - 改用 `React.lazy` 或在組件內部動態 `import()` 函式庫。
  - 只有在 Markdown 中偵測到對應的程式碼區塊（如 ```mermaid）時，才啟動下載。
- **MathJax**：
  - 將 `MathJaxContext` 移出 `App.tsx` 的全域包裹，改為按需載入。

### 1.2 匯出工具延遲載入
- **xlsx / pdf-lib / html2canvas**：
  - 這些套件僅在使用者點擊「匯出」或「列印」時才需要。
  - 改為在按鈕點擊事件中使用動態 `import()`。

---

## 🚀 第二階段：優化主執行緒與渲染效能

### 2.1 修正強制自動重排 (Forced Reflow)
- **優化捲動同步邏輯 (`rebuildLineMap`)**：
  - 目前在迴圈中頻繁讀取 `getBoundingClientRect()`。
  - 改為使用 `Promise.all` 或批次處理，先收集所有座標資訊，再更新狀態。
- **動畫優化**：
  - 檢查側邊欄與 Modal 動態，確保優先使用 `transform` 與 `opacity` 而非變動 `width`/`left`。

### 2.2 構建優化
- **Vite Target**：將 `build.target` 設置為 `esnext` 或現代瀏覽器版本，減少 Babel helpers 與不必要的 Polyfills。

---

## ♿ 第三階段：無障礙 (Accessibility) 與細節修正

### 3.1 補全 ARIA 標籤
- 為所有無文字按鈕（如關閉、縮放、切換模式）補上 `aria-label`。
- 為數字輸入框（如 Zoom）加上關聯的 `<label>`。

### 3.2 快取策略 (Nginx/Server)
- 如果適用，在 `nginx.conf` 或部署環境中加入更強的 `Cache-Control` 標頭。

---

## 📋 實作優先順序
1. **P0 (即刻執行)**：圖表套件 (Mermaid, Vega) 與 數學公式 (MathJax) 的延遲載入。
2. **P1 (關鍵效能)**：修正 `App.tsx` 中的 Forced Reflow。
3. **P1 (易用性)**：補全無障礙標籤。
4. **P2 (進階)**：匯出工具延遲載入與構建目標調整。

---

## 🧪 驗證方式
- 使用 Chrome DevTools 的 `Lighthouse` 進行前後對比。
- 使用 `Network` 面板觀察初始 JS 下載量。
- 使用 `Performance` 面板監測捲動時的 Frame Rate 與 Main Thread Activity。

### 效能問題核心整理

|類別 | 問題重點 | 影響指標 |
| :--- | :--- | :--- |
| JS 執行量 | 1,159 KiB 未使用的程式碼，導致主執行緒繁忙達 4.4 秒。 | TBT (1.4s), SI (3.9s) |
| 資源阻塞 | mermaid, mathjax, xlsx 等大型套件一次性載入。 | LCP (2.5s) |
| 渲染卡頓 | 頻繁觸發 強制自動重排 (Forced Reflow)。 | 交互響應速度 |
| 無障礙 | 按鈕與輸入框缺少 aria-label 或標籤。 | Accessibility Score |

### 具體優化建議方案

1.  第一階段：程式碼分割與懶加載 (最有效)
* 這是目前最嚴重的問
  - 動態匯入 (Dynamic Imports)：不要在首頁就載入 mermaid, mathjax, xlsx。應改為當使用者切換到對應分頁或輸入特定語法時，再透過 import() 動態加載。

  - 外部 CDN 優化：目前的 mathjax 從 JSDelivr 載入，雖然方便，但其解析時間也計入了你的 TBT。考慮改用更輕量化或自定義打包的版本。

2. 第二階段：優化主執行緒負擔
  - Web Workers：Markdown 的解析 (Parsing) 與圖表的渲染是計算密集型任務。建議將這些邏輯移至 Web Worker 執行，避免 UI 凍結。

  - 減少重複轉換：報告指出偵測到 舊版 JavaScript (Babel helpers)。若你的目標使用者大多使用現代瀏覽器，請調整打包配置（如 Vite/Webpack 的 target），移除不必要的 Polyfills。

3. 第三階段：修正 DOM 作業與渲染
  - 批次更新 DOM：針對報告提到的「強制自動重排」，應檢查 index-CkHN9kJo.js 中是否有在迴圈裡讀取 offsetWidth 或 getBoundingClientRect 後立刻修改樣式的行為。應改為「先集體讀取，再集體寫入」。

  - CSS 合成 (Compositing)：部分動畫（如側邊欄）應優先使用 transform 與 opacity，避免觸發重排。

4. 第四階段：無障礙與細節修正
* 標籤補全：

  - 為所有 <button> 加入 aria-label="開啟側邊欄" 或 aria-label="功能介紹"。

  - 為縮放比例的 <input> 補上隱藏的 <label>。

  - 快取策略：為靜態資源 (CSS/JS) 設定更長的 Cache-Control: max-age=31536000。

預期成效: 若成功實施 懶加載 與 程式碼分割，預計可以減少約 60% 的初始 JS 載入量，效能分數有望從 41 分躍升至 80 分以上。
